# Release Notes Curation Protocol

You are helping curate raw release notes into polished GitHub release notes for
marimo. The file `release_notes.md` has already been generated by
`scripts/generate_release_notes.py`. Your job is to research PRs, identify
highlights, fix formatting, and iterate with the user until the notes are
publish-ready.

---

## Prerequisites

- `release_notes.md` exists in the repo root (generated by `scripts/generate_release_notes.py`)
- `gh` CLI is authenticated and available

If `release_notes.md` is missing, tell the user to run the generation script
first:

```sh
uv run scripts/generate_release_notes.py <previous-tag>
```

---

## Workflow

Follow these steps in order. Each step involves user interaction — present
findings, get confirmation, then proceed.

### Step 1: Read current notes and recent releases

1. Read `release_notes.md` from the repo root
2. Fetch the 3 most recent GitHub releases for format/style reference:
   ```sh
   gh release view <tag> --json body
   ```
3. Note the established format conventions:
   - One-liner intro paragraph summarizing the release theme
   - `## ⭐️ Highlights` section with `###` subheadings per highlight
   - Screenshots use `<p align="center"><img width="600" src="..." /></p>`
   - Doc links where relevant: `[Read the docs](https://docs.marimo.io/...)`
   - Category sections: Enhancements, Bug fixes, Documentation, Preview features
   - Contributors section with new contributor callouts

### Step 2: Research PR details for highlight candidates

For each item in the Enhancements section (and any `release-highlight` labeled
items), fetch PR details:

```sh
gh pr view <number> --json title,body,labels
```

Look for:
- **Labels**: `release-highlight`, `enhancement`, `preview`, `includes-media`
- **PR body**: Screenshots, demos, descriptions of user-facing impact
- **Scope**: How significant is the change for end users?

Prioritize **stable features over experimental/preview ones** for highlights.

### Step 3: Propose highlights to user

Present 3-5 ranked highlight candidates. For each, include:
- PR title and number
- One-sentence summary of why it's highlight-worthy
- Whether it has screenshots/media in the PR body
- Whether docs exist for it

Ask the user:
- Which candidates to include as highlights
- Whether to add/remove any
- Preferred ordering

### Step 4: Categorization check

Review all entries and flag any that may be miscategorized:
- Experimental/preview features listed under Enhancements (should be in Preview)
- Bug fixes that are really enhancements (or vice versa)
- Items that should be removed (internal-only changes that slipped through)
- Items that appear in multiple sections

Present findings and ask the user to confirm moves.

### Step 5: Link and formatting audit

Check every entry for:
- `(#1234)` references that aren't wrapped as proper GitHub links —
  should be `([#1234](https://github.com/marimo-team/marimo/pull/1234))`
- Bare commit SHAs that should link to the commit
- Consistent bullet formatting (`* Title ([#N](url))`)
- Conventional commit prefixes that weren't stripped (e.g., `feat:`, `fix:`)

Fix all issues automatically — no need to ask for each one.

### Step 6: Write the highlights section

Build the `## ⭐️ Highlights` section. For each chosen highlight:

1. Create a `### Descriptive title` subheading (user-friendly, not the PR title)
2. Write a 1-3 sentence description of what it does and why it matters
3. Include a screenshot from the PR body if available, using:
   ```html
   <p align="center">
     <img width="600" src="URL" />
   </p>
   ```
4. Include code examples if the feature has a clear CLI or API surface
5. Link to docs if available: `[Read the docs](https://docs.marimo.io/...)`

Also write a one-liner intro paragraph at the top of the notes summarizing the
release theme (before the highlights section).

### Step 7: Assemble and present

Write the complete curated `release_notes.md` with this structure:

```markdown
## What's Changed

<one-liner intro summarizing the release theme>

## ⭐️ Highlights

### <Highlight 1 title>

<description>

<screenshot if available>

### <Highlight 2 title>

...

## ✨ Enhancements
* ...

## 🐛 Bug fixes
* ...

## 📚 Documentation
* ...

## 🔬 Preview features
* ...

## Contributors
Thanks to all our community and contributors who made this release possible: @user1, @user2, ...

And especially to our new contributors:
* @newuser made their first contribution in https://github.com/marimo-team/marimo/pull/NNNN

**Full Changelog**: https://github.com/marimo-team/marimo/compare/old...new
```

### Step 8: Iterate

Present the full draft to the user. Go back and forth on:
- Highlight wording, ordering, and screenshots
- Whether to add/remove/swap highlights
- Intro paragraph tone
- Any remaining formatting issues

Keep iterating until the user is satisfied.

## Key Conventions

These are derived from past marimo releases. Follow them precisely.

### Highlight format

```markdown
### User-friendly feature title

1-3 sentences describing the feature and its value to users.

<p align="center">
  <img width="600" src="https://github.com/user-attachments/assets/..." />
</p>

Optional code example or CLI usage.

[Read the docs](https://docs.marimo.io/...) for more details.
```

### Entry format

```markdown
* Entry title ([#1234](https://github.com/marimo-team/marimo/pull/1234))
```

- Strip conventional commit prefixes (`feat:`, `fix:`, `docs:`, etc.)
- Capitalize the first letter after stripping
- Do not include author names in line items

### Image markup

Always use centered HTML, not markdown image syntax:

```html
<p align="center">
  <img width="600" src="URL" />
</p>
```

Adjust width as appropriate (400-650 typical range).

### Contributors section

```markdown
## Contributors
Thanks to all our community and contributors who made this release possible: @user1, @user2

And especially to our new contributors:
* @newuser made their first contribution in https://github.com/marimo-team/marimo/pull/NNNN
```

New contributors are identified by checking if they have prior merged PRs:

```sh
gh pr list --author <username> --state merged --limit 2
```

If they only have 1 merged PR (the one in this release), they're new.

---

## Guardrails

- **Always ask before publishing.** Never run `gh release create` — only prepare
  the markdown. The user will publish manually.
- **Preserve all entries.** Don't remove items from category sections unless the
  user explicitly asks. You may move items between sections with confirmation.
- **Don't invent information.** Descriptions should come from PR bodies and docs.
  If you're unsure what a feature does, fetch the PR details or ask the user.
- **Highlights should be user-facing.** Internal improvements, refactors, and
  dependency bumps are not highlight material, even if they're large changes.
